# kreuzberg-surrealdb v0.1.0 — The Feature Story

> A zero-dependency, 5-stage RAG pipeline component.
> kreuzberg extracts, chunks, and embeds. SurrealDB stores and searches. This connector is the glue.

---

## The Pipeline

```
File → Extract → Chunk → Embed → Store → Search → LLM-ready chunks
       kreuzberg  kreuzberg  kreuzberg  connector  SurrealDB
```

Five stages. Stages 1-3 happen inside a single kreuzberg call. Stage 4 is the connector mapping kreuzberg's output into SurrealDB's schema. Stage 5 is SurrealDB doing what databases do — the connector wraps it in simple Python methods.

---

## Stage 0: Connect and Set the Stage

Before anything flows through the pipeline, the database needs to be ready.

**Connect to SurrealDB** — Point at a running instance via WebSocket, or spin up an in-memory embedded instance for development. No external server needed for testing.

**Auto-create the schema** — One `setup_schema()` call creates everything: two tables (`documents` and `chunks`), all typed fields, vector indexes, full-text indexes, a custom analyzer, and uniqueness constraints. The schema is created with enforced types (SCHEMAFULL) before any data exists — standard database practice. Fields are defined, tables are empty, ready for data.

**Map embedding dimensions automatically** — The user picks a kreuzberg embedding preset ("compact" at 384d, "balanced" at 768d, "large" at 1024d). The connector reads the dimension from the preset and configures the HNSW vector index to match. The user never thinks about dimension numbers.

> 4 features. Result: a database shaped and ready to receive documents.

---

## Stage 1: Extract

kreuzberg takes a file — any file — and pulls out clean text and rich metadata.

**Ingest any file** — Hand `ingest_file()` a path. kreuzberg auto-detects the format (PDF, DOCX, XLSX, PPTX, EML, EPUB, HTML, images — 75+ formats) and extracts text, metadata, quality score, detected languages, and keywords. One call to kreuzberg's `extract_file()` handles the entire extraction pipeline including chunking and embedding.

**Ingest entire directories** — `ingest_directory()` with a glob pattern. kreuzberg batch-extracts in parallel using its Rust core. The connector batch-inserts documents and chunks into SurrealDB. Handles partial failures — if one file fails extraction, the rest still go through.

**Skip duplicates** — Before extracting, the connector hashes the file bytes with SHA-256. If the hash or source path already exists in the database, it skips the file entirely. No wasted extraction on files already ingested.

> 3 features. Result: raw files turned into structured extraction results.

---

## Stage 2: Chunk

kreuzberg splits extracted text into retrieval units — chunks. This is the most critical stage for search quality.

**Chunking is always on.** The unit of search is the chunk, not the document. A 50-page PDF as one string is too long for LLM context windows, full of irrelevant content, and impossible to cite precisely. Chunks solve all three problems.

**Automatic chunking with metadata** — kreuzberg splits text with configurable strategy (semantic or character-based), size, and overlap. Each chunk carries positional metadata: byte offsets, chunk index, token count, page range. Semantic strategy (the default) respects topic boundaries — each chunk is a coherent unit of meaning.

**Page range per chunk** — Each chunk records the first and last page it spans. Enables precise citations: "Source: report.pdf, pages 5-7." Not all formats have pages (text files, HTML store null).

> Chunking is part of the extraction call, not a separate step. But conceptually it's the stage that determines retrieval granularity — the single biggest lever for search quality.

---

## Stage 3: Embed

kreuzberg generates vector embeddings for each chunk — locally, on CPU, with zero API calls.

**Local embedding generation** — Every chunk gets a vector embedding via ONNX/FastEmbed. No API keys. No network calls. No GPU required. The "balanced" preset produces 768-dimensional vectors. These embeddings are what make semantic search possible — they capture meaning, not just keywords.

**Embeddings are on by default.** They can be toggled off with `embed=False` for BM25-only use cases. When off, chunks store text without vectors. Vector and hybrid search become unavailable — only keyword search works.

> Embedding also happens inside the same kreuzberg extraction call. The vectors come back attached to each chunk.

## Stage 4: Store

The connector maps kreuzberg's extraction output into SurrealDB's typed schema. This is where the glue does its work.

### Chunk Storage

The connector takes each chunk from kreuzberg and inserts a record into the `chunks` table:

- `content` — the chunk text (indexed for BM25)
- `embedding` — the vector (indexed for HNSW)
- `document` — record link to parent document
- `chunk_index` — position in the sequence
- `char_start`, `char_end` — character offsets in the original text
- `token_count` — for LLM context window budgeting
- `first_page`, `last_page` — for citation

### Document Storage

The connector stores one record per ingested file in the `documents` table with everything kreuzberg extracted:

- `source` — file path (unique, used for deduplication)
- `content` — full extracted text
- `content_hash` — SHA-256 of file bytes (unique, used for deduplication)
- `mime_type` — detected format
- `title`, `authors`, `created_at` — common metadata as typed top-level fields
- `metadata` — flexible object for format-specific fields (PDF page count, Excel sheet names, email headers, EXIF data)
- `quality_score` — kreuzberg's extraction quality assessment (float, filterable at search time)
- `detected_languages` — array of language codes
- `keywords` — extracted keywords with scores (YAKE/RAKE)
- `ingested_at` — auto-set by SurrealDB on insert

### Indexes

Three indexes and one analyzer, created by `setup_schema()`:

| Index | Type | Purpose |
|---|---|---|
| `idx_chunk_embedding` | HNSW vector | Semantic search over chunks |
| `idx_chunk_content` | BM25 full-text | Keyword search over chunks |
| `idx_doc_source` | UNIQUE | Deduplication by source path |

| Analyzer | Purpose |
|---|---|
| `chunk_analyzer` | class tokenizer + lowercase + snowball(english) for BM25 stemming |

> Result: kreuzberg's rich extraction output is structured, typed, indexed, and ready for search.

## Stage 5: Search

SurrealDB does the searching. All three search modes are **SurrealDB-native capabilities** — HNSW, BM25, and RRF fusion exist in SurrealDB regardless of this connector. kreuzberg's role is already done: it provided the vectors and text that SurrealDB searches over.

The connector's job at search time is two things:
1. **Embed the query string** — SurrealDB needs a vector to do vector search, but it doesn't generate embeddings. kreuzberg only generates embeddings during chunking (stage 3), not for standalone text. So the connector uses FastEmbed directly — the same ONNX engine kreuzberg uses internally — to embed the query string. Same model, same dimensions, consistent results.
2. **Wrap SurrealDB queries** into simple Python methods.

### Three search methods

**Hybrid search** — The default `search()` method. Runs three operations in a single SurrealDB round-trip:
1. Vector KNN search via HNSW (finds "return credit process" when you query "refund policy" — semantic match, zero keyword overlap)
2. BM25 full-text search (finds "the refund policy is in section 4.2" — exact keyword match a vector might rank lower)
3. Reciprocal Rank Fusion combining both rankings (a chunk that ranks high in BOTH is almost certainly relevant)

This is the headline feature. Research is unambiguous — hybrid outperforms either mode alone.

**Pure vector search** — `vector_search()` for embedding-only semantic retrieval. Query embedded locally, KNN search against HNSW index.

**Pure keyword search** — `fulltext_search()` for traditional BM25 keyword matching. No embeddings involved.

### Search parameters

- `limit` — how many results to return (default 10)
- `min_quality` — filter out poorly extracted documents (quality score threshold)

### Result format

Results come back as a list of dicts with `content`, `document` reference, `chunk_index`, and relevance score. Designed for `r["content"]` access — ready to inject straight into an LLM prompt.

> 6 features. Result: three ways to retrieve the most relevant chunks.

## Configuration Philosophy

### The 4 knobs that affect search quality

Out of 20 configurable dimensions across the pipeline, only 4 meaningfully change retrieval results. Everything else has correct defaults a RAG specialist wouldn't touch.

| Knob | What it controls | Where it lives | Default |
|---|---|---|---|
| **Chunk size + overlap** | Retrieval unit granularity — completeness and precision | Constructor | 512 / 64 |
| **Embedding preset** | Semantic resolution — how well meaning is captured | Constructor | "balanced" (768d) |
| **Analyzer language** | BM25 stemming correctness — must match corpus | Constructor | "english" |
| **Quality threshold** | Noise floor — filter bad extractions from results | Search method | None (no filter) |

### Constructor design

```python
ingester = DocumentIngester(
    # Connection (required plumbing)
    url="ws://localhost:8000",
    username="root",
    password="root",
    namespace="rag",
    database="documents",

    # Retrieval tuning (the 4 knobs — all have defaults)
    embedding_preset="balanced",    # "compact" (384d) | "balanced" (768d) | "large" (1024d)
    chunk_size=512,                 # characters per chunk
    chunk_overlap=64,               # overlap between chunks
    analyzer_language="english",    # snowball stemmer language
)
```

### Zero-config path

```python
ingester = DocumentIngester(url="ws://localhost:8000")
await ingester.setup_schema()
await ingester.ingest_file("document.pdf")
results = await ingester.search("What is the refund policy?")
```

Four lines. Semantic chunking, balanced embeddings, cosine distance, english analyzer, hybrid search. All defaults.

### Advanced configuration

For the rare user who needs to tune HNSW parameters (EFC, M), BM25 parameters (k1, b), distance metric (COSINE/EUCLIDEAN/MANHATTAN), RRF k, or batch insert size — these exist in an advanced config object. They're available. They're not prominent.

## What This Is Not

- **Not a graph connector** — SurrealDB has native graph capabilities (`RELATE`, edge traversal), but this version doesn't use them. Graph features are v0.2.0 scope.
- **Not a cache** — SurrealDB is not a Redis replacement. No TTL, no pub/sub.
- **Not a SurrealDB v3.0 connector** — Targets v2.x syntax. v3.0 support is a future option.
- **Not for massive scale out of the box** — HNSW indexes are in-memory in SurrealDB. They rebuild on server restart and memory scales linearly with vector count.
- **Not a document-level search engine** — The unit of search is always the chunk. Users who want full-text search over whole documents without chunking don't need this package — kreuzberg + 20 lines of SurrealDB code covers that.

## The 25 Features

| Stage | What happens | Count | kreuzberg's role | SurrealDB's role |
|---|---|---|---|---|
| 0: Setup | Connect, create schema, map dimensions | 4 | Provides embedding preset dimensions | Hosts the schema and indexes |

**kreuzberg is the crawler.** It turns files into structured, embedded data.
**SurrealDB is the engine.** It indexes, stores, and searches that data.
**This connector is the glue.** `setup_schema()`, `ingest_file()`, `search()`.
